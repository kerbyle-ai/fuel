# Тикет в поддержку Timeweb Cloud (готово к вставке)

**Куда:** панель Timeweb Cloud → Поддержка → Новое обращение  
**Тема:** kvmnvm-939 — SSH недоступен с интернета (таймаут / обрыв на KEX)

---

## Текст обращения

Здравствуйте!

Не могу подключиться по SSH к облачному серверу **kvmnvm-939**.

**Публичный IPv4:** 147.45.175.194  
**Публичный IPv6:** 2a03:6f00:a::2:b276  
**Проект:** fuel-map (развёртывание приложения «Карта топлива»)

### Симптомы

1. С рабочей станции (Windows, VPN отключён): клиент OpenSSH выдаёт  
   `Connection to 147.45.175.194 port 22: Connection timed out`  
   (иногда соединение устанавливается, но сессия не доходит до аутентификации).

2. С другой точки проверки:
   - **TCP 22:** соединение устанавливается (SynAck), задержка ~10 ms.
   - **SSH-баннер:** приходит `SSH-2.0-OpenSSH_9.6p1 Ubuntu-3ubuntu13.16`.
   - **Полный SSH-клиент:** после обмена версиями обрыв по таймауту на этапе **KEX** (после `SSH2_MSG_KEXINIT`), до ввода пароля не доходит.
   - **TCP 8090:** отказ в соединении (сервис не слушает — ожидаемо до деплоя).
   - **IPv6 на порт 22:** таймаут (IPv4 предпочтителен).

3. ICMP ping до 147.45.175.194 с части сетей не отвечает (не критично, если SSH должен работать по TCP).

4. Облачный Firewall (FWaaS): прошу подтвердить, что к **kvmnvm-939** привязана группа с **разрешающими** правилами для входящего **TCP 22, 80, 443, 8090** с 0.0.0.0/0 (и при необходимости ::/0).

### Что уже пробовали на стороне клиента

- Отключение VPN и прокси.
- `ssh -4`, увеличение `ConnectTimeout`.
- Проверка с мобильного интернета (рекомендуем повторить у себя при блокировке порта 22 у провайдера).

### Просьба выполнить на стороне Timeweb / на ВМ

1. Проверить, что на **kvmnvm-939** запущен и слушает порт 22 процесс **sshd** (`systemctl status ssh`, `ss -tlnp | grep :22`).

2. Проверить **локальный firewall** на ВМ (ufw/iptables/nftables), что **22/tcp** разрешён для входящих.

3. Выполнить с вашей стороны (консоль / serial), если SSH с нашей сети недоступен, официальный скрипт включения serial-консоли:  
   `wget -qO- https://st.timeweb.com/cloud-static/scripts/serial_enable.sh | bash`  
   и сообщить, когда serial-консоль доступна в панели.

4. Проверить отсутствие блокировок на уровне **гипервизора / anti-DDoS / сетевых ACL**, из-за которых после SSH-баннера не проходит key exchange.

5. При необходимости — перезапуск `ssh`: `systemctl restart ssh`.

6. Сообщить результат проверки **с вашей тестовой машины** командой  
   `ssh -v root@147.45.175.194`  
   (достаточно факта: доходит ли до `Authenticated`).

### Цель

Нужен стабильный **SSH по IPv4:22** для деплоя Docker-приложения (порты 80/443/8090 после настройки).
