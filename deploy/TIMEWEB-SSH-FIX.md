# SSH на Timeweb VPS: «Connection timed out during banner exchange»

Диагностика для **kvmnvm-939**, IP **147.45.175.194** (fuel-map). Симптомы: ping OK, TCP 22/8090 «открыты», SSH обрывается на banner exchange, HTTP на 8090 — таймаут без ответа.

---

## Что показала проверка (факты)

| Проверка | Результат |
|----------|-----------|
| TCP к порту 22 | Соединение устанавливается (SynAck) |
| Баннер SSH (сырой сокет) | **Иногда приходит:** `SSH-2.0-OpenSSH_9.6p1 Ubuntu-3ubuntu13.16`, задержка **~1,2–1,4 с** |
| OpenSSH (Windows) | Часто: `Connection timed out during banner exchange` (при том же IP) |
| Порт 8090 | TCP OK, HTTP `GET /api/health` — **0 байт ответа** (зависший фронт/nginx или перегруженный хост) |
| Порт 12345 (случайный) | Не «настоящий» сервис — при массовых проверках тоже был TCP без данных (типично для перегрузки/фильтра) |
| IPv6 SSH с ПК | `Permission denied` на уровне ОС клиента (IPv6 с вашей машины недоступен) |
| Альтернативные порты 2222, 22022 | TCP «открыт», баннера SSH нет — **не отдельный sshd** |
| Маршрут с вашего ПК | Трафик идёт через **sing-tun VPN** (`172.19.0.1`) — может усугублять таймауты SSH |

**Вывод:** проблема **не в «выключенном SSH после ребута»**. На сервере **работает OpenSSH 9.6 (Ubuntu)**, но ответы **медленные и нестабильные**. Это типично для:

1. **Перегрузки VPS** (CPU/RAM, swap, OOM killer душит `sshd` / docker).
2. **Забитого диска** (`/` или `/var` 100%) — sshd и docker встают.
3. **Зависшего Docker Compose** (порт 8090 слушает «оболочку», API/nginx не отвечают).
4. **Агрессивного UFW/iptables** или облачного файрвола Timeweb (реже даёт именно banner timeout, чаще reset).
5. **Клиентского VPN/провайдера** — TCP проходит, прикладной обмен рвётся; проверка с **мобильного интернета без VPN** обязательна.

Ребут **не лечит** переполненный диск, OOM и «залипший» docker после `up --build`.

---

## Гипотеза первопричины (по приоритету)

1. **VPS перегружен или диск заполнен** → sshd отвечает с задержкой; Windows OpenSSH не дожидается баннера; параллельно контейнеры на 8090 не отдают HTTP.
2. **Облачный файрвол / DDoS-защита Timeweb** в режиме «под атакой» → нестабильные сессии (реже, если баннер всё же виден с сокета).
3. **VPN на вашем ПК (sing-tun)** → расхождение: сырой Python иногда видит баннер, `ssh.exe` — нет.

## Что проверить в панели Timeweb Cloud

Путь может слегка отличаться в новом UI; ищите по названиям.

### 1. Карточка сервера → Сеть / Firewall / «Правила»

- Должно быть **разрешено входящее TCP 22** с `0.0.0.0/0` (или ваш IP, если ограничиваете).
- Разрешите **80, 443** (для будущего nginx + Let’s Encrypt).
- Для теста деплоя — **8090** (в production потом лучше закрыть снаружи и оставить только 80/443).
- **Описание в UI:** таблица правил, колонки «Порт», «Протокол», «Источник», «Разрешить/Запретить». Убедитесь, что нет правила «запретить всё» выше allow для 22.

### 2. Безопасность / DDoS / «Защита от атак»

- Если включён режим **Under attack** / жёсткая фильтрация — **смягчите** или добавьте свой IP в allowlist.
- **Описание:** переключатель «Защита L3–L7», «Базовый / Усиленный» — на время настройки поставьте «Базовый».

### 3. VNC / консоль (главный обход, если SSH с ПК нестабилен)

- **Серверы** → ваш VPS → **Консоль** / **VNC**.
- Логин: `root`, пароль из панели (или сброс пароля в карточке сервера).

### 4. Снимки и переустановка (крайний случай)

- Если в консоли `sshd` не стартует и диск забит — **расширьте диск** или **переустановите Ubuntu 22.04** (с потерей данных на диске, если нет бэкапа).

## Однострочная диагностика в VNC (вставить в консоль)

```bash
echo "=== load/disk ==="; uptime; df -h / /var; free -h
echo "=== ssh ==="; systemctl status ssh --no-pager; ss -tlnp | grep ':22'
echo "=== docker ==="; docker ps -a 2>/dev/null; curl -m5 -sS http://127.0.0.1:8090/api/health || echo FAIL
echo "=== oom ==="; dmesg -T 2>/dev/null | tail -20 | grep -i oom || true
```

Если `df` показывает **100%** — очистите логи/docker (`docker system prune -af`, `journalctl --vacuum-size=200M`), затем `systemctl restart ssh docker`.

## Порт 8090 «открыт», но пустой ответ

Это **отдельно от SSH**:

- Слушатель на 8090 есть (docker nginx), но **upstream (api) не отвечает** или идёт бесконечный build.
- В VNC: `cd /opt/fuel-map && docker compose -f docker-compose.yml -f docker-compose.prod.yml ps` и `logs -f api nginx`.
- Проверка только с сервера: `curl -sS http://127.0.0.1:8090/api/health` → ожидается `{"status":"ok"}`.

Пока health с localhost не OK — снаружи 8090 тоже будет «висеть».

## Чеклист «сделано — SSH заработал»

- [ ] В облачном файрволе Timeweb открыт TCP 22 (и при необходимости 80/443/8090).
- [ ] `df -h /` < 90%, `free -h` есть свободная RAM/swap.
- [ ] `systemctl is-active ssh` → `active`.
- [ ] С сервера: `curl -m5 http://127.0.0.1:8090/api/health` OK.
- [ ] С ПК **без VPN**: `ssh root@147.45.175.194` показывает запрос пароля (не banner timeout).
- [ ] UFW на сервере: `ufw status` — не блокирует 22 с вашего IP.
